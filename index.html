Blind SQL Injection

SQL盲注

描述

SQL盲注是一类请求数据库正确或错误的问题并根据基于应用程序的返回来决定答案的SQL注入攻击。虽然这类攻击经常用在被配置显示普通错误信息的web应用中，但这并没有缓解容易被SQL注入的代码问题。
当攻击者利用SQL注入，有时候web应用显示数据库返回的SQL查询语法不正确的错误信息。盲注几乎等同于正常的SQL注入，唯一不同的是从数据库获取数据的方式。当数据库并不支持输出数据到web页面，攻击者被迫通过询问数据库一系列正确的或错误的问题来取回数据。这使得利用SQL注入漏洞更加困难，但是并不是不可能的…

实例

攻击者可能通过几种方式识别发送的请求返回是否正确或错误：

基于内容

使用简单页面，通过ID参数展示文章，攻击者可能执行一些简单的测试来判定页面是否容易受到SQL注入漏洞攻击。

示例URL:

http://newspaper.com/items.php?id=2

发送如下查询到数据库:

SELECT title, description, body FROM items WHERE ID = 2

攻击者然后可能尝试注入返回false的查询:

http://newspaper.com/items.php?id=2 and 1=2

现在SQL查询看起来像是这样：

SELECT title, description, body FROM items WHERE ID = 2 and 1=2

如果web应用容易受到SQL注入，那么它有可能不返回任何东西。为了确认，攻击者将注入返回true的查询：

http://newspaper.com/items.php?id=2 and 1=1

如果页面内容返回true是不同于返回false的页面，那么当执行的查询返回正确或错误，攻击者就能识别了。

一旦这个被确认了，唯一的限制就是数据库管理员设置的不同于SQL语法，和攻击者的想象能力的权限了。

基于时间

这类盲注依赖于数据库暂停一段指定的时间，然后返回结果，则表明SQL查询成功执行。使用这个方法，攻击者使用如下的逻辑枚举每个期望获取的数据片段字母：

如果数据库名字的第一个字母是A，等待10秒。

如果数据库名字的第一个字母是B，等待10秒。

…

Microsoft SQL Server

http://www.site.com/vulnerable.php?id=1' waitfor delay '00:00:10'--

MySQL

SELECT IF(expression, true, false)

使用一些耗时的操作，比如

远程数据库指纹

当攻击者能够确定他的查询什么时候返回正确和失败，那么它可能是关系型数据库的指纹。这将会使得整个攻击更加简单些。如果使用基于时间的方式，这有助于确定使用的哪一类数据库。另一个比较常见的方法是去调用返回当前时间的函数。MySQL，MSSQL，和Oracle有不一样的函数实现这个，各自为now(),getdate(),sysdate()。

Cross-site Scripting

概述

跨站脚本攻击（XSS）是一类注入，其恶意脚本被注入到可信的网站中。攻击者使用应用程序发送恶意代码，一般是在浏览器端脚本的表格中，给不同的终端用户的时候XSS攻击就发生了。缺陷使得这些攻击得逞非常普遍，并且发生在任何一处使用来自用户的输入和输出而有没有校验或编码它的web应用程序中。

攻击者可以利用XSS发送恶意脚本给信任的人。终端用户的浏览器没有办法了解到脚本是否该被信任，并且将会执行脚本。因为它认为脚本来自于可信的源，恶意的脚本能够读取到任意cookies,session tokens,或者浏览器和被网站使用过的保留的敏感信息。这些脚本甚至能够重写HTML页面的内容。更多关于不同类型的XSS漏洞详情，看这里：Types of Cross-Site Scripting

描述

跨站脚本攻击（XSS）攻击发生在：

  1. 数据通过不信任的源输入到应用程序，在web请求中更频繁。
  2. 数据包含动态的内容被送到web用户，没有经过恶意内容的校验。

恶意的内容发送到浏览器经常包含一段js，也可能包含HTML，Flash，或任何浏览器可能执行的代码。大量基于XSS的攻击都是无限制的，但是他们通常包括传递隐私数据，像cookies或者其他session信息，对于攻击者来说，将受害者重定向到被攻击者控制的页面，或者存在漏洞的网站的幌子下在用户机器上执行恶意的操作

存储和反射XSS攻击

XSS攻击大致是能被分为两类：存储和反射。还有个第三类，没前两类那么有名的XSS攻击，被叫做DOM XSS。

存储XSS攻击

存储攻击是指注入的脚本被永久的存储在目标服务器，如数据库，消息论坛，访客日志，评论区域等。然后当受害者请求被存储的信息时候，获取到服务器的恶意脚本。存储XSS有时候也指持久或者Ⅰ类XSS。

反射XSS攻击

反射攻击是那些注入脚本被web服务器反弹，比如在错误消息，搜索结果，或者任何包含一些或所有输入作为部分请求被发送到服务器而返回。反射攻击通过其他路由传递给受害者，比如在一封邮件信息，或者一些其他的web网站。当一个用户被欺骗去点击一个恶意的链接，提交一个特殊构造的表单，或者甚至浏览一个恶意的网站，被注入的代码就传递到了有漏洞的网站，反射攻击返回到用户的浏览器。浏览器然后执行代码，因为它来自于一个被信任的服务器。反射XSS有些时候是值非持久或者Ⅱ类XSS。

其他类的XSS漏洞

除了存储和反射XSS，另外一类XSS，DOM XSS在2005年被Amit Klein提出。OWASP推荐在OWASP文章里描述的XSS分类： Types of Cross-Site Scripting,覆盖了所有的XSS条目，并把他们组织成存储与反射XSS和服务端与客户端XSS，其中DOM XSS是客户端XSS的子集。

XSS攻击后果

XSS攻击后果是相同的无论是否是存储的还是反射的（或者基于DOM的）。差别在于载荷是怎么到达服务器的。不要错误的认为“只读”或者“brochureware”网站是不会受到严重的反射XSS攻击的。XSS能引起多种范围从一个烦恼程度到账户妥协的问题给终端用户。最严重的XSS攻击设计用户的会话cookie信息的披露，允许攻击者戒指用户会话并接管账户。其他有害的攻击包括披露终端用户文件，安装木马程序，重定向用户到一些其他页面和站点或者修改展示的内容。XSS漏洞允许攻击者修改新闻稿或新闻项目，这可能影响公司股票价格或使消费者丢失信心。在医药网站的XSS漏洞可能允许攻击者修改药量信息导致过计量。关于这类攻击详细信息，请参见Content_Spoofing

如何保护你自己

对XSS的主要防御在OWASP XSS Prevention Cheat Sheet_Prevention_Cheat_Sheet)里已经描述了。

另外，你在所有支持HTTP TRACE的web服务器上关闭它是很重要的。攻击者能够通过javascript盗取cookie数据，尽管在客户端当document.cookie被禁用或不支持的时候。攻击在一个用户发表一段恶意脚本到论坛的时候被挂起，当另一个用户点击这个连接，一个从服务器收集用户cookie信息的异步HTTP 跟踪调用就被触发了，然后将它发送到另一个恶意服务器去收集cookie信息，因此攻击者能够进行session劫持攻击。这个通过在web服务器上移除HTTP TRACE很容易减少攻击。

OWASP的ESAPI项目已经产生了一套多种语言的可重复使用的安全组件，包括校验和转义程序以防止参数被篡改和XSS注入攻击。此外，OWASP的WebGoat培训项目应用有跨站脚本和数据编码的课程。

SQL injection

概述

SQL 注入攻击是由来自客户端对应用程序插入或者注入SQL查询组成的。一次成功的SQL注入攻击能够从数据库读取到敏感数据，修改数据库数据（插入/更新/删除），在数据库上执行管理员操作（比如关闭数据库管理系统），恢复数据库管理系统上的给定文件，并在某些条件下向操作系统发出命令。SQL注入攻击以注入攻击的一种，其SQL命令被注入到数据层为了使预定义的SQL命令执行产生预期效果。

威胁模型

  • SQL注入攻击允许攻击者欺骗身份，篡改已经存在的数据，造成比如交易无效的或者改变结余的不可否认的问题。还允许完整的披露系统上的所有数据，毁坏数据或者要不然就让它不可用，并且成为数据库服务器的管理员。
  • PHP和ASP应用一些老的函数接口使用得很普遍，使得其SQL注入非常常见。由于编程接口的本质使用合理，J2EE和ASP.NET应用程序不太有可能被SQL注入利用的。
  • SQL注入攻击的严重程度是受到攻击者的技能和想象力限制的，在适当程度上采取纵深防御的措施，比如说以低权限连接数据库服务器等，总的来说，把SQL注入的严重程度认为有很大影响。

描述

SQL注入错误发生在：

  1. 数据由来源不可信的地方进入程序。
  2. 数据被用来动态的拼接SQL查询。

主要的后果有：

  • 机密性： 由于SQL数据库一般保存了敏感数据，SQL注入漏洞使得其失去机密性是很常见的问题。
  • 认证： 如果比较低级SQL命令被用来检查名字和密码，它将有可能通过一个以前不知道密码的另一个用户连接到系统。
  • 授权： 如果授权信息是保存在SQL数据库里面的，那么它可能通过成功利用一个SQL注入漏洞修改这个授权信息。
  • 完整性： 就像它可能读取到敏感信息一样，它也可能通过一次SQL注入攻击修改或者删除这个信息。

危险因素

可能影响的平台：

  • 语言： SQL
  • 平台： 任意（需要和数据库交互）

SQL注入已经成为数据库驱动的web网站的常见问题。这个问题很容易被检测到，而且很容易利用，正是因为如此，任意网站或者最小用户群体的软件包都很有可能受到这种攻击尝试。

从本质上来讲，攻击是通过插入元字符到输入的数据中，然后将原来不存在的SQL命令放置在控制面板，这个缺陷实质上取决于SQL使得数据层和逻辑业务层没有真正的区别。

例子

实例1

SQL语句：

select id, firstname, lastname from authors

如果有人输入：

Firstname: evil'ex
Lastname: Nevman

查询字符串就变成：

select id, firstname, lastname from authors where forename = 'evil'ex' and surname = 'newman'

数据库将尝试如此运行：

Incorrect syntax near il' as the database tried to execute evil.

一个用Java编码的SQL语句安全版本如下：

String firstname = req.getParameter("firstname");
String lastname = req.getParameter("lastname");
// FIXME: do your own validation to detect attacks
String query = "SELECT id, firstname, lastname FROM authors WHERE forename = ? and surname = ?";
PreparedStatement pstmt = connection.prepareStatement( query );
pstmt.setString( 1, firstname );
pstmt.setString( 2, lastname );
try
{
    ResultSet results = pstmt.execute( );
}

实例2

下面是c#的代码的动态构造和执行的SQL语句，来查询匹配特殊名字的项。这个查询仅展示那些和当前认证用户用户名相匹配的项目。

    ...
string userName = ctx.getAuthenticatedUserName();
string query = "SELECT * FROM items WHERE owner = "'" 
                + userName + "' AND itemname = '"  
                + ItemName.Text + "'";
sda = new SqlDataAdapter(query, conn);
DataTable dt = new DataTable();
sda.Fill(dt);
...

这个查询代码目的是执行如下：

SELECT * FROM items
WHERE owner = 
AND itemname = ;

然而，由于这个查询是由一连串的基础字符串和用户输入的字符串动态构造的，所以这个查询仅仅在项目名次并不包含单引号字符的时候才会表现正确。假如一个攻击者输入name' or 'a'='a作为用户名，那么查询就会变成如下：

SELECT * FROM items
WHERE owner = 'wiley'
AND itemname = 'name' OR 'a'='a';

添加的OR 'a'='a'正好造成那一列用于是真值，所以这个查询变得逻辑上等同于更简单的查询：

SELECT * FROM items;

查询的简化使得攻击者绕过查询仅返回被认证的用户所拥有的条目的限制；现在查询返回所有的存储在项目表里的条目，忽视他们特殊的所有者。

实例3

这实例检查不同的恶意值传给实例1中被构造和执行的查询的效果。假如黑客攻击者输入name');DELETE FROM items;--作为项目名，那么查询就变成如下两个查询了：

SELECT * FROM items 
WHERE owner = 'hacker'
AND itemname = 'name';

DELETE FROM items;

--'

很多数据库服务器，包括微软的SQL Server 2000,允许多条SQL语句通过分号分隔后被一次执行。虽然在Oracle和其他不允许批量执行由分号分隔的语句的数据库服务器上批量执行的攻击字符串会导致错误，但是在一些允许批量执行命令的服务器上，这种类型的攻击允许攻击者对数据库执行任意命令。注意到成对的连字符（–），在大多数数据库服务器中被声明作为注释，而且不被执行的。在这种情况下，注释字符的作用是从被修改过的查询中除去最后一个单引号。在不支持这种注释方式的数据库中，一般的攻击仍然能够有效的使用一个类似实例1中展示的技巧。假如一个攻击者输入字符串name'); DELETE FROM items; SELECT * FROM items WHERE 'a'='a,下面三个有效的语句将会被创建：

SELECT * FROM items 
WHERE owner = 'hacker'
AND itemname = 'name';

DELETE FROM items;

SELECT * FROM items WHERE 'a'='a';

一个防止SQL注入攻击的传统方法是把他们当作一个输入校验的问题来处理，要么接收安全值白名单中的唯一字符，要么识别并避开那些潜在的恶意数据黑名单。白名单可以是强制执行严格的输入检查规则的非常有效的手段，但是参数化的SQL语句需要更好的维护成本就能提供更安全的保障。黑名单策略几乎总是充满了漏洞，使得其无法有效的防止SQL注入攻击，比如说，攻击者能够：

  • 黑名单未包含的目标
  • 寻找方法去绕过某些转义元字符的限制
  • 使用存储过程来隐藏注入的元字符

在输入给SQL查询的时候手工转移字符是有帮助的，但是它不会让你的应用免遭SQL注入攻击。

另一个普遍建议的处理SQL注入攻击的方案是使用存储过程，尽管存储过程防止了一些类型的SQL注入攻击，他们不能防止其他很多类型。例如，下面的PL/SQL程序跟第一个实例容易受到一样的SQL注入攻击。

procedure get_item (
    itm_cv IN OUT ItmCurTyp,
    usr in varchar2,
    itm in varchar2)
is
    open itm_cv for ' SELECT * FROM items WHERE ' ||
            'owner = '''|| usr || 
            ' AND itemname = ''' || itm || '''';
end get_item;

存储过程通过限制传递给语句的参数类型，有助于防止SQL注入攻击。然而，有很多限制方法的局限性和很多有趣的语句仍然能够被传递到存储过程。再次强调，存储过程可以防止一些攻击，但是他们不能完全的使你的应用系统抵御SQL注入攻击。
